### §2.2 Vercel handleUpload → GCS Signed POST Policy 行为映射

| 阶段 | 当前 `@vercel/blob/client.handleUpload` | GCS 等价（手实现） | 关键差异 |
|---|---|---|---|
| **Browser 发请求** | POST `/api/upload`，body `{type: "blob.generate-client-token", payload: {...}}` | POST `/api/upload`，body `{type: "generate-signed-url", pathname, contentType, clientPayload?}` | shape 不同 → 由 `upload-client.ts` (b-3) 重写 fetch 调用；**b-2 内 backward-compat 不必要**（同 chain push） |
| **Server: validate body** | handleUpload 内部解析 body type | facade `parseUploadRequest()` zod schema 验 + 分流 2 phase | facade 显式 schema (新 export `SignedUploadRequestSchema`) |
| **Server: clientPayload 验证** | handleUpload 调 caller `onBeforeGenerateToken(pathname, clientPayload)` | facade 直接调 `policy.clientPayloadSchema.safeParse(req.body.clientPayload)` | 等价（同步 vs callback），lifecycle 简化 |
| **Server: gen token / signed URL** | handleUpload 内部 mint token (Vercel 私有 JWT) | facade 调 `bucket.file(finalKey).generateSignedPostPolicyV4({ expires, conditions, fields })` → `[{ url, fields }]` | GCS 返 `url + fields` (browser 用 multipart/form-data POST)；与 PUT signed URL 不同（**A 决策**） |
| **Server: addRandomSuffix** | handleUpload 内部 append | facade `finalKey = addRandomSuffix ? key + "-" + uuid8() : key`；写入 policy `conditions: [["eq", "$key", finalKey]]` | **server-side enforce** finalKey；browser 不可改 |
| **Server: 返 envelope** | `{type: "blob.generate-client-token", clientToken: "..."}` | `{type: "signed-upload-policy", url, fields, completionToken}` | shape 不同，**UploadEnvelope brand 保护**，caller (b-3 shim) 仅 fetch().json() pass to upload-client，不破坏 facade contract |
| **Browser: upload** | 用 token PUT 到 Vercel Blob | 用 fields + file multipart POST 到 `url` (GCS 直接接收) | b-3 重写 browser shim |
| **Server: completion notify** | Vercel webhook 调回 caller `onUploadCompleted({blob: {url, pathname, contentType}})` | browser 上传完成后 POST `/api/upload`，body `{type: "completion", completionToken, blobInfo: {url, pathname, contentType, size?}}` → facade verify HMAC token + 调 `policy.onCompleted` | **GCS 无 webhook**；ping 是 client-driven（不可信） → 必须 HMAC token 防 tamper（**C 决策**） |

### §2.3 设计决策点（5+，本 scope 核心）

#### A. Signed URL 形式：POST Policy vs PUT Signed URL

| 选项 | 描述 | 取舍 |
|---|---|---|
| **A1**（W1 倾向）| `file.generateSignedPostPolicyV4({expires, conditions, fields})` 返 `{url, fields}`；browser 用 `multipart/form-data` POST | ✅ 可 enforce `["content-length-range", 0, maxBytes]` server-side（防恶意上传超 200MB）；✅ 可 enforce `["eq", "$Content-Type", "video/mp4"]` 或 `["starts-with", "$Content-Type", "video/"]`（与 `allowedContentTypes` 1:1）；✅ 可 enforce `["eq", "$key", finalKey]`（server-side addRandomSuffix 保护）；⚠️ browser 端需手拼 multipart FormData（b-3 改动） |
| A2 | `file.getSignedUrl({version:"v4", action:"write", contentType, expires})` 返单字符串 URL；browser 单 PUT body | ✅ browser 端简单（单 PUT）；❌ 不能 enforce content-length（attacker 上传 10GB 用满 quota）；❌ contentType 只能签时 fix 一个值（无法 enforce multiple allowed）；❌ key 由 URL 路径锁定，但 addRandomSuffix 必须 client 协商 round-trip（多一次 RTT） |
| A3 | Hybrid: PUT URL + server pre-flight HEAD verify content-length | ❌ pre-flight HEAD 多一次 RTT；attacker 已浪费 bandwidth 才被 reject；UX 退化 |

**W1 倾向：A1 POST Policy**。理由：viral-reviewer 上传 200MB video / 100MB PDF / 30MB BGM，**server-side content-length-range 是 mandatory 安全防线**。A2 PUT signed URL 无法关 quota abuse。

#### B. Completion 通知机制：browser ping vs GCS event vs poll

| 选项 | 描述 | 取舍 |
|---|---|---|
| **B1**（W1 倾向）| Browser POST 完成后 ping 同 endpoint，body `{type:"completion", completionToken, blobInfo}`；facade verify token + 调 `policy.onCompleted` | ✅ 与现 `policy.onCompleted` lifecycle 1:1；✅ 不需 GCS Pub/Sub 配置（避免 W2 P5.2.4 input 依赖）；✅ ping 1 次 RTT 内完成；⚠️ ping 来自 untrusted client 必须 HMAC token 防 tamper（C 决策） |
| B2 | GCS Object Finalize event → Pub/Sub topic → Cloud Run subscriber → facade onCompleted | ✅ 100% 可信（GCS-driven）；❌ Pub/Sub topic + subscription + Cloud Run subscriber endpoint 全新 infra → W2 P5.2.4 deploy.yml 必须 frozen 后才能验；❌ at-least-once 语义意味着 onCompleted 必须 idempotent（当前 caller `console.log` 是 ok，未来 DB 写入需重 audit） |
| B3 | facade poll `bucket.file(finalKey).exists()` until true (with timeout) | ❌ 服务端 hold connection 直到 GCS 写完 → 撑爆 Cloud Run 并发 quota；❌ Vercel 300s timeout 复现回归 |

**W1 倾向：B1 browser ping**。理由：B2 增加 W2 deploy infra 依赖（违反 §2.6 ownership-dep "lib code can start now" 原则）；B3 是反 pattern。B1 + HMAC token 是最小可信 emulation。

#### C. Completion Token 防 tamper 机制

| 选项 | 描述 | 取舍 |
|---|---|---|
| **C1**（W1 倾向）| HMAC-SHA256(`UPLOAD_SIGNING_SECRET` env, `JSON.stringify({finalKey, contentType, maxBytes, expiresAt})`)，token 拼 base64url(payload) + "." + base64url(hmac)；facade verify 时检 expiresAt > now、finalKey 与 blobInfo.url match | ✅ stateless（无 KV/Redis 依赖）；✅ tamper 立即 fail；✅ expiresAt 防 replay；⚠️ 新 env `UPLOAD_SIGNING_SECRET` 32-byte random — W2 P5.2.4 Secret Manager 注入 |
| C2 | Redis/KV 存 token → finalKey 映射，verify lookup | ❌ 引入新 infra (KV)；❌ 跨 Cloud Run instance 一致性问题；❌ b-2 引入新依赖打破 "lib code can start now" |
| C3 | 不验，trust browser ping | ❌ attacker 任意触发 onCompleted（当前 hook 仅 console.log 看似无害，但 a-3 followup 已 mandate "future caller 可能 DB 写"）；❌ 安全审计立即 fail |

**W1 倾向：C1 HMAC stateless**。理由：viral-reviewer 当前 caller 仅 console.log，但 W3 `cd7f45a` D3 推翻明确"未来 DB 写需 break clean"，必须现在就建可信信道，避免后续追溯。

**Secret 生成**：runbook §5 GCP Secret Manager step append "create `upload-signing-secret`: `openssl rand -hex 32`"。本 scope 仅 docs 提及，secret 实际创建是 W2 cloud-side ops。

#### D. Per-upload TTL 策略

| 选项 | 描述 | 取舍 |
|---|---|---|
| **D1**（W1 倾向）| POST policy `expires = Date.now() + 60*60*1000`（60min，与 W3 `12b3b18` C write TTL 一致）；completion token `expiresAt = Date.now() + 60*60*1000` | ✅ 一致 60min，覆盖 200MB video 慢网络上传（4G 5MB/s ≈ 40s，但用户可能 stall）；✅ 与 frozen W3 write TTL 1:1 |
| D2 | POST 15min / completion 15min | ❌ 200MB on 1MB/s 上传 ≈ 200s 接近，无 retry buffer；❌ 不一致 W3 frozen |
| D3 | POST 60min / completion 24h | ❌ completion 24h 让 attacker 有大窗口 replay；不必要长 |

**W1 倾向：D1 60min/60min**。与 frozen 一致 + 覆盖慢网络。

#### E. Error mapping（保留 13 case baseline + 新增 GCS-specific）

| 现 error code | 状态 | 触发条件（b-2 后） |
|---|---|---|
| `invalid_upload_body` | **保留** | json parse fail / SignedUploadRequestSchema reject / clientPayloadSchema reject |
| `signed_upload_failed` | **保留** | `generateSignedPostPolicyV4` 抛 / GCS auth fail / completion endpoint internal error |
| `storage_not_configured` | **复用 b-1 code** | `getStorage().enabled === false`（GCS_BUCKET_NAME 缺）；route 层 503 by handler |
| `provider_specific` (b-1 case) | **保留** | facade re-throw 已有 StorageError 子类 |
| **新 `completion_token_invalid`** | **NEW** | HMAC verify fail（tamper / wrong secret） |
| **新 `completion_token_expired`** | **NEW** | `expiresAt < now` |
| **新 `completion_blob_mismatch`** | **NEW** | completion body 的 `blobInfo.url` 不含 finalKey（attacker 伪造 blobInfo） |

**3 个新 code 显式 ack**：与 b-1 `url_not_in_bucket` 新增模式一致；caller 不需新 catch 分支（仍走 `signed_upload_failed` 500 UX，code+cause log 进 console.error）。

#### F. UploadEnvelope 内部 shape 变化

**契约**（不变）：opaque brand type，caller 必须 `NextResponse.json(envelope)` 直接 pass，不可 destructure。

**内部 shape**（变）：
- 现：`{type: "blob.generate-client-token", clientToken: "<JWT>"}` (Vercel 私有)
- 后：`{type: "signed-upload-policy", url: string, fields: Record<string, string>, completionToken: string, finalKey: string}` 或 `{type: "completion-ack", success: true}`

**brand 保护机制**：`UploadEnvelope = { readonly [_uploadEnvelopeBrand]: never }` 保留；TypeScript 编译期阻止 caller 看内部字段（`as unknown as UploadEnvelope` 强 cast 是唯一途径，与 a-4 同 pattern）。

**TS compile-time 守门**（参考 b-1 commit 3a 大嘉奖 pattern）：
```ts
// tests/storage/signed-upload.test.ts 末尾
type _NoFieldsOnUploadEnvelope = "fields" extends keyof UploadEnvelope ? never : true;
const _PROOF_UPLOAD_ENVELOPE_OPAQUE: _NoFieldsOnUploadEnvelope = true;
void _PROOF_UPLOAD_ENVELOPE_OPAQUE;
```

#### G. addRandomSuffix server-side enforce

GCS POST policy `conditions: [["eq", "$key", finalKey]]` 把 finalKey 锁死。browser 无法替换 `$key` form field（GCS 拒绝），server-side `finalKey = key + "-" + crypto.randomUUID().slice(0,8)` 唯一 source of truth。

与 b-1 commit 2 `put()` 内部 addRandomSuffix 等价 pattern；test 必须 case："policy conditions 含 `[\"eq\", \"$key\", finalKey]` 且 finalKey 含 8-char hex suffix when addRandomSuffix=true"。

    ```ts
    const body = await req.json().catch(() => throw InvalidUploadBodyError);
    const parsed = SignedUploadRequestSchema.safeParse(body);
    if (!parsed.success) throw new InvalidUploadBodyError(...);

    if (parsed.data.type === "generate-signed-url") {
      // phase 2a: clientPayload validate
      const cpResult = policy.clientPayloadSchema.safeParse(parsed.data.clientPayload);
      if (!cpResult.success) throw new InvalidUploadBodyError(...);
      // phase 2b: addRandomSuffix
      const finalKey = policy.addRandomSuffix
        ? `${parsed.data.pathname}-${crypto.randomUUID().slice(0,8)}`
        : parsed.data.pathname;
      // phase 2c: contentType allowlist
      if (!policy.allowedContentTypes.includes(parsed.data.contentType)) {
        throw new InvalidUploadBodyError(`contentType ${parsed.data.contentType} not in allowlist`);
      }
      // phase 2d: gen policy + sign completion token
      try {
        const {url, fields} = await generateSignedPostPolicy(finalKey, {
          contentType: parsed.data.contentType,
          maxBytes: policy.maxBytes,
          expiresMs: 60 * 60 * 1000,
        });
        const completionToken = signCompletionToken({finalKey, contentType: parsed.data.contentType, maxBytes: policy.maxBytes});
        return {type: "signed-upload-policy", url, fields, completionToken, finalKey} as unknown as UploadEnvelope;
      } catch (e) { /* re-throw StorageError / wrap as signed_upload_failed */ }
    }

    if (parsed.data.type === "completion") {
      // phase 3a: verify token
      const payload = verifyCompletionToken(parsed.data.completionToken); // throws completion_token_invalid|expired
      // phase 3b: verify blobInfo.url contains finalKey
      if (!parsed.data.blobInfo.url.includes(encodeURI(payload.finalKey))) {
        throw new StorageError("completion_blob_mismatch", ...);
      }
      // phase 3c: invoke onCompleted (swallow + log per D3)
      if (policy.onCompleted) {
        try { await policy.onCompleted(parsed.data.blobInfo); }
        catch (hookErr) { console.error(`[${policy.logTag}] onCompleted hook failed:`, hookErr); }
      }
      return {type: "completion-ack", success: true} as unknown as UploadEnvelope;
    }
    ```
