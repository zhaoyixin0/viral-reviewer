## §2.2 Caller wiring path（每 caller 升级路径表）

> 模板 §2.2 原表是 "URL 来源 → preset"（phase 2 防误用 preset）；phase 3.5 已确定 preset（phase 2.5 sample-verify 完工），本表 adapt 为 "URL 来源 → sync check / checkAsync / fetchWithAllowlist 升级路径"。

| # | Caller | URL 来源 | 当前 preset | 当前调用 | phase 3.5 升级 | DNS rebinding 攻击面 |
|---|---|---|---|---|---|---|
| 1 | `prepareAssets` | client JSON body `videoUrls[]` + `bgmUrl` | VERCEL_BLOB_PRESET | `for url: opts.urlAllowlist.check(url); throw if !ok; fetch(url)` | `for url: await opts.urlAllowlist.checkAsync(url); throw if !ok` + `videoUrls.map(url => fetchWithAllowlist(url, opts.urlAllowlist))` | 中（Vercel Blob CDN host 可信,DNS resolve 第二次漂移 in-band 攻击概率低,但 fetch 重 resolve 仍漏） |
| 2 | `extractFramesAndAudio` | derived from #6/#7 callers | VERCEL_BLOB_PRESET / TIKTOK_INSTAGRAM_CDN_PRESET | sync check + `fetch(videoUrl)` + writeFile + ffmpeg 读本地 | sync check 保留 fast-fail + `fetchWithAllowlist(videoUrl, opts.urlAllowlist)` 替 `fetch`；ffmpeg 不动（已读本地 /tmp）| 高（TT/IG CDN URL 来源 user-controlled scrape, 攻击者可控 host） |
| 3 | `template-brief` blob loader | client JSON body `blobUrl` | VERCEL_BLOB_PRESET | inline `URL_ALLOWLIST.check(blobUrl)` + `fetch(blobUrl)` | `fetchWithAllowlist(blobUrl, URL_ALLOWLIST)` + 既有 `UrlAllowlistError` mapping（已有的 console.warn + 400 模式扩 dns/private reason）| 中 |
| 4 | `technique-match` batch loader | client JSON body `videoUrls[]` | VERCEL_BLOB_PRESET | pre-stream `for url: urlAllowlist.check(url)` + in-stream `fetch(url)` | pre-stream `await Promise.all(videoUrls.map(url => urlAllowlist.checkAsync(url)))` + in-stream `fetchWithAllowlist` | 中 |
| 5 | `account-profile` frame analyze | Apify scrape `top1.videoDownloadUrl` | TIKTOK_INSTAGRAM_CDN_PRESET | 继承 #2 via `analyzeAccountTopVideo` | 继承 #2 | **高**（攻击者可注册 TT 账号注 evil host） |

**关键决策点（W2 → W3 verdict needed）**：

### 决策 A：批量 checkAsync 是否需 caller 内并发优化

- **A1** 串行 `for url of urls: await checkAsync(url)`
  - ✅ 实现简单,错误归属清晰（第一个 fail 即返回）
  - ❌ N 个 URL 串行 DNS resolve → latency N × ~30ms
- **A2** 并行 `Promise.all(urls.map(checkAsync))`
  - ✅ Latency 单次 ~30ms
  - ❌ 失败时 promise.all 短路,需 `Promise.allSettled` 才知所有失败原因（也许 caller 只需第一个 fail）
- **A3** 并行 `Promise.allSettled` + 收集所有 deny reason
  - ✅ 完整诊断信息
  - ❌ 实现略复杂,多数 caller 不需

**W2 倾向 A2** for `prepareAssets` / `technique-match` 批量场景（latency 优于诊断完整性,失败时 throw 第一个 deny URL 已够）。单 URL caller（`template-brief`）用单调 `await fetchWithAllowlist` 即可。

**请 W3 拍板**：A2 接受还是要求 A3 完整诊断？

### 决策 B：ffmpeg.ts 是否同时保留 sync check

- **B1** 完全替换（仅 `fetchWithAllowlist`,sync check 删）
  - ✅ 代码更简洁
  - ❌ 失去 fast-fail（DNS resolve 之前先拒 invalid_url / scheme_denied / host_denied），增~30ms latency on rejected case
- **B2** sync check 保留 fast-fail + `fetchWithAllowlist`（双重 check; checkAsync 内部 sync check 也做一遍 = triple check）
  - ✅ Fast-fail 保 latency
  - ❌ 同 URL 校验 3 次（sync 入口 + checkAsync 内 sync + checkAsync 内 DNS）—— 浪费但不破

**W2 倾向 B1** —— `fetchWithAllowlist` 内部 checkAsync 已含 sync check fast-fail（前 4 个 reason 走 sync path,不发 DNS),无需 caller 额外加；checkAsync 短路是 phase 3 commit 2 已实现 "sync deny 不调 DNS" 行为。

**请 W3 拍板**：B1 接受？

### 决策 C：caller error mapping `dns_resolve_failed` vs `resolved_private_ip`

phase 3 lib `UrlAllowlistError` 已扩 `resolvedIp` / `cause`。caller 层 error mapping：

| Reason | HTTP status | Server log level | Response body | Client retry |
|---|---|---|---|---|
| `invalid_url` / `scheme_denied` / `host_denied` / `private_ip` | 400 `url_denied` | `console.warn` | 不暴露 reason 详情 (phase 2 verdict B2) | ❌ |
| `dns_resolve_failed` | **502 `dns_resolve_failed`** + `Retry-After: 5` header | `console.warn` | 暴露 transient 标识（caller 可重试） | ✅ |
| `resolved_private_ip` | **400 `url_denied`** (与 host_denied 同 response 防探测) | **`console.error`** + 记 `resolvedIp` | 不暴露 reason (防 SSRF probe) | ❌ |

**W2 倾向 C-above mapping** —— 区分 transient 故障可重试 vs security event 不可重试,server log 区分 warn / error。

**请 W3 拍板**：mapping 表接受？还是要求 `resolved_private_ip` 不发 retry header 即可,console.error 太强？

### 决策 D：测试 fixture 升级策略（§4 #3 防御）

既有 caller test fixture 多 mock `fetch` 而非 `checkAsync` DNS resolve。phase 3.5 升级要求：

- **D1** 每 test file 加 `vi.mock("node:dns")`（dns-resolve.test.ts 模式）
- **D2** 加 helper `tests/__stubs__/dns-mock.ts` 复用统一 mock
- **D3** 测 caller 时 mock `lib/url-allowlist/dns-resolve.ts` 的 `safeResolveIp` 直接（不走真 dns）

**W2 倾向 D2** —— 4-5 个 caller test file 共享 helper,maintenance 集中。

**请 W3 拍板**：D2 接受？
