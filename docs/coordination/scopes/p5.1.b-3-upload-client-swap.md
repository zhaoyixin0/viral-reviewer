## §1 Applicability

P5.1.b-2 merged 后，server 端 lifecycle 已 GCS 化 (signed POST policy + HMAC completion token + 11-code StorageError set)。Browser 端 `upload-client.ts` 当前仍走 Vercel SDK 的 `upload` 函数，与 b-2 server 端 contract 不兼容 — **此 mismatch 当前是 production-broken**（b-2 server expects `{type:"generate-signed-url",...}`, Vercel client sends `{type:"blob.generate-client-token",...}`）。

b-3 是 GCS migration 最后一块 lib 改动，完成后 4 frontend callers 与 GCS 端到端连通；b-4 即可移除 `@vercel/blob` dep。

### 已 frozen 设计前置（不重审）

- b-2 server lifecycle contract (W3 verdict 78b7d2f)：2-phase request shapes (`generate-signed-url` / `completion`) + opaque `UploadEnvelope` + HMAC completion token + GCS POST policy `{url, fields}` shape
- HMAC payload canonical pipe-concat 5 fields (b-2 c1)
- `pathname` + `contentType` 不含 `|` schema regex 守门 (b-2 c2 HIGH fix)
- `urlToKey` 严格 bucket+key match for completion verify (b-2 c2 HIGH fix per nit #2)

### §2.2 Vercel `upload` → hand-rolled GCS POST 行为映射

| 阶段 | 当前 `@vercel/blob/client.upload` | b-3 hand-rolled | 差异 |
|---|---|---|---|
| **API surface** | `upload(pathname, body, opts) → Promise<PutBlobResult>` | 同 signature → `Promise<{url, pathname, contentType, size}>` | shape 形状一致 (caller `.url` access 不变) |
| **opts 字段** | `{access, handleUploadUrl, contentType, clientPayload?, onUploadProgress?}` | 同 (onUploadProgress 取决于决策 C) | — |
| **Phase 1: gen token** | POST handleUploadUrl `{type:"blob.generate-client-token",...}` → token JSON | POST handleUploadUrl `{type:"generate-signed-url", pathname, contentType, clientPayload}` → UploadEnvelope `{url, fields, completionToken, finalKey}` | shape 不同；b-3 与 b-2 server 端配合 |
| **Phase 2: upload** | Vercel internal: PUT 到 Blob with token | 手 build FormData（含所有 `fields` + 最后 file body）+ POST 到 envelope.url (GCS direct) | multipart/form-data；GCS 拒绝 if conditions violated (size > maxBytes, contentType mismatch, key mismatch) |
| **Phase 3: completion** | Vercel webhook 触发 server onUploadCompleted | 客户端 POST handleUploadUrl `{type:"completion", completionToken, blobInfo:{url, pathname, contentType, size}}` → server 验 token + 触发 onCompleted | 完整由 client driven (无 webhook 替代) |
| **Return shape** | `{url, pathname, contentType, contentDisposition, size?}` (PutBlobResult) | `{url, pathname, contentType, size}` (subset — `contentDisposition` Vercel-only) | callers 只 access `.url` 与少数 `.pathname` |
| **Error model** | throws on non-2xx with Vercel-specific error class | throws `UploadError` (new) with `{code, message, status?, response?}` | unified error shape；caller try/catch 不需改 |
| **Progress reporting** | XHR-based byte-level percentage callback | **决策 C: drop / phase-only / XHR-based byte-level** | 影响 BriefUploader UI |

### §2.3 设计决策点（核心 8 个）

#### A. API surface 兼容程度

| 选项 | 描述 | 取舍 |
|---|---|---|
| **A1**（W1 倾向）| 保 `upload(pathname, body, opts) → Promise<{url, pathname, contentType, size}>` 与 Vercel SDK 1:1 兼容 | ✅ 4 callers 零改动 (除 BriefUploader progress UI 取决于 C)；✅ 与 BlobInfo subset 形状一致；⚠️ `contentDisposition` 字段丢失 — 但 callers 都不用 (grep verified) |
| A2 | 改 `upload(req: UploadRequest) → Promise<UploadResult>` 用 explicit shape | ❌ 4 callers 都需改 (3 lines × 4 = 12 改动)；❌ 与 PutBlobResult 不兼容打破回退路径 |
| A3 | export `uploadToGCS` 新函数 + `upload` 作 alias to A1 form | ❌ 双 export 误用风险；YAGNI |

**W1 倾向：A1**。

#### B. Error model

| 选项 | 描述 | 取舍 |
|---|---|---|
| **B1**（W1 倾向）| 引入 `UploadError extends Error { code: string; status?: number; response?: unknown }` — 浏览器侧 mirror of server-side `StorageError` shape | ✅ caller 可 switch 区分 retryable (5xx) / non-retryable (4xx)；✅ 错误码透明 (`gen_signed_url_failed` / `gcs_upload_failed` / `completion_ping_failed` / `network`)；⚠️ 新 export type — caller 需手 import for switch (optional) |
| B2 | throw `new Error(message)` plain (current Vercel 模式) | ❌ caller 无法区分 retry policy；❌ 与 server-side StorageError 不对称 |
| B3 | throw raw `Response` object | ❌ awkward API；callers 需 `await error.json()` 难用 |

**W1 倾向：B1**。

`UploadError.code` 集合：
- `gen_signed_url_failed`：phase 1 POST 非 2xx 或 schema 不符
- `gcs_upload_failed`：phase 2 POST 到 GCS 非 2xx (含 403 policy violation / 400 bad form)
- `completion_ping_failed`：phase 3 POST 非 2xx
- `network`：phase 1/2/3 fetch reject (offline / DNS / TLS)
- `aborted`：caller cancelled via opts.signal (AbortSignal)

#### C. Progress reporting （BriefUploader 依赖）

| 选项 | 描述 | 取舍 |
|---|---|---|
| **C1**（W1 倾向）| **Phase-only progress**: callback fires 3 次 `({phase:"signing"|"uploading"|"completing"})`；删 `percentage` 字段 | ✅ 简单 (无需 XHR)；✅ 满足 BriefUploader 显示 "上传中..." 类基本 UX；⚠️ BriefUploader UI 改 (10 行；display phase label instead of percentage bar) |
| C2 | **XHR-based byte-level**: 用 `XMLHttpRequest` 替换 fetch for phase 2 (upload to GCS)，订阅 `upload.onprogress` 事件 → `({percentage, bytes, total})` | ✅ caller 零改动 (percentage 行为保留)；❌ XHR 模式 fetch 模式分裂 (phase 1/3 用 fetch，phase 2 用 XHR)；❌ XHR 错误处理不如 fetch；❌ XHR 不支持 AbortSignal natively (需手 wire abort listener) |
| C3 | **No progress callback support**：删 `onUploadProgress` opt，warn at runtime if passed | ❌ 破坏 BriefUploader UX (显示 "上传中..." 但无进度) — 100MB PDF 上传时 user 觉得 hang；❌ silent contract break |

**W1 倾向：C1 phase-only**。理由：
- viral-reviewer BriefUploader 是少数 callers (4 中 1)，其它 3 个不用 progress
- Phase-only 满足 UX 基本需求 (用户知道在干什么)
- 字节级 percentage 是 nice-to-have 不是 critical (b-3 优先 ship)
- XHR 引入 architectural complexity > value

BriefUploader UI 改动：原 `{ percentage }` → `{ phase }`，进度条改成 phase label ("正在签名" / "上传中" / "完成中") + indeterminate spinner。

#### D. clientPayload 传递

| 选项 | 描述 | 取舍 |
|---|---|---|
| **D1**（W1 倾向）| 透传 opts.clientPayload (任意 JSON-serializable value) 到 phase 1 body | ✅ 当前所有 callers 不传 clientPayload (server 强制 z.null())；✅ 与 b-2 server schema 1:1 |
| D2 | 强制 stringify (Vercel SDK 模式：clientPayload 是 string) | ❌ 多余 stringify 步骤；server schema 已 accept unknown |
| D3 | 不传 (drop entire feature) | ❌ 破坏 forward-compat hook (server schema 已 accept) |

**W1 倾向：D1**。

#### E. AbortSignal 支持

| 选项 | 描述 | 取舍 |
|---|---|---|
| **E1**（W1 倾向）| opts.signal?: AbortSignal — pass to 3 fetch calls；abort → throw UploadError("aborted") | ✅ caller 可 cancel 大文件上传；✅ 与 fetch 标准 API 一致 |
| E2 | 不支持 abort | ❌ 200MB video 上传到一半用户切换页面时无法 cancel — request stays in-flight wastes bandwidth |

**W1 倾向：E1**。callers 当前不用 signal — 引入是 forward-compat。

#### F. FormData 字段顺序

GCS POST policy 文档要求：**fields 必须先 append，file 必须最后 append**。否则 GCS 拒 `400 Bad Request: Missing required field 'key'`。

| 选项 | 描述 | 取舍 |
|---|---|---|
| **F1**（W1 倾向）| `Object.entries(envelope.fields).forEach(([k,v]) => fd.append(k, v)); fd.append("file", body);` — 显式 ordering | ✅ 标准 GCS 模式；✅ test 验顺序 |
| F2 | for-of with explicit ordering check | ✅ same outcome；冗余 |

**W1 倾向：F1**。

#### G. Test environment

| 选项 | 描述 | 取舍 |
|---|---|---|
| **G1**（W1 倾向）| 用 `// @vitest-environment jsdom` per-file pragma (upload-client.test only)；vitest.config 不全局改 | ✅ 不影响其它 server-side tests 性能；✅ 隔离；⚠️ 加 dep `jsdom` 或 `happy-dom` |
| G2 | vitest.config global `environment: 'jsdom'` | ❌ 53 个其它 test file 启 jsdom 拖慢 |
| G3 | 不用 jsdom — mock `FormData` / `fetch` 全 node-native | ⚠️ Node 18+ 原生有 FormData + fetch (no need for jsdom)；test 写法接近真实但要 `globalThis.fetch = vi.fn()` setup；✅ 最轻 |

**W1 倾向：G3 (Node-native, no jsdom dep)**。理由：Node 18+ runtime 原生提供 FormData + fetch + Blob，upload-client.ts 实现只用这些 web API；no DOM APIs needed (no document/window/element)。最轻量。

#### H. CLIENT_WHITELIST 处理

| 选项 | 描述 | 取舍 |
|---|---|---|
| **H1**（W1 倾向）| 完全删除 `CLIENT_WHITELIST` set + `CLIENT_IMPORT` regex 检查 (空 whitelist 后无用) | ✅ 简化 check-storage-imports.ts；✅ tripwire 仍 effective via TOP_WHITELIST (`@vercel/blob`) — b-4 会进一步移除 TOP_WHITELIST | 
| H2 | 保 CLIENT_WHITELIST 为空 set 作 tripwire | ❌ 永远 empty set，自动 reject 任何 import — 与 H1 行为相同但代码更冗；prefer H1 simplicity |
| H3 | 全 b-3 不动 CLIENT_WHITELIST，留 upload-client.ts 在 set 内 (stale entry) | ❌ 违反 b-2 c3 precedent (stale entry 立即清) |

**W1 倾向：H1**。同时 update index.ts docstring。
