## §1 Applicability

P5.8 是 Cloud Run platform-native observability foundation：把 viral-reviewer 既有 `console.warn/error`（Vercel Functions 默认 plain-text logs）切到 **structured JSON logs**，对接 Cloud Logging 的 `severity` + `jsonPayload` 字段。Cloud Logging 自动 parse JSON stdout 行（per [Cloud Run docs: Structured logging](https://cloud.google.com/run/docs/logging#run_manual_logging)）。

**为什么需要**：
- ✅ Cloud Logging dashboard 按 `severity` 过滤 (`severity>=WARNING`) + 按 `jsonPayload.module` 分组
- ✅ Stack trace 完整保留（plain `console.error(e)` 截断 toString，结构化 `{error: { message, stack }}` 完整）
- ✅ 关键 metadata 显式（`{module: "trending/fetch", op: "tiktokStage2", err_class: "TimeoutError"}`）便于查询
- ✅ Cloud Run 文档明示推荐 structured logging 配 Log-based Metrics + Alerts

按 scope-template §1 强制走全套 §2 必填栏（多文件改动 + cross-cutting refactor）。

**不在本 scope**：
- ❌ 引入 OpenTelemetry / distributed tracing（Cloud Trace integration 是 P5.8 之后的独立 scope candidate）
- ❌ 引入 Sentry / 外部 error tracking SaaS（成本 + Vercel→Cloud Run 平迁不引入新 SaaS）
- ❌ 改造 `console.log`（仅 `warn/error`；info 级别日志当前不影响 ops，留 follow-up）
- ❌ Log-based Metrics + Alerts 配置（cloud-side ops，defer user 回来）

### §2.3 设计决策点（A-F）

#### A. Logger API shape — singleton vs factory

| 选项 | 描述 | 取舍 |
|---|---|---|
| **A1**（W4 倾向） | `export const logger = createLogger({ default_module?: string })` factory + per-module instance：`const log = createLogger({ module: "trending/fetch" })`；call site `log.warn(msg, ctx)` | ✅ module 字段 once-binding，call site 不重复传；✅ 测试可注入 mock；✅ 与 pino/winston API 一致（未来切实际 logger lib 零改动 caller） |
| A2 | 单 singleton `logger.warn(msg, ctx)`；call site 每次传 `{module: "trending/fetch"}` | ✅ 简单；❌ module 字段重复，易写错；❌ 难 mock per-module |
| A3 | 用 pino 直接 import（不写 helper） | ❌ +npm 依赖；❌ 大量 pino 配置 boilerplate；❌ Cloud Logging severity 映射需自己 patch；P5 phase 焦点是平迁不是引入新 deps |

**W4 倾向：A1**。factory pattern 兼容未来替换底层 logger（pino/winston/opentelemetry），caller 不动；module 字段 once-binding 简洁。

**请 W3 拍板**：A1 OK？还是 A2 简单优先？

#### B. Severity mapping (Cloud Logging spec)

| 选项 | 描述 |
|---|---|
| **B1**（W4 倾向） | `logger.warn → severity: "WARNING"`; `logger.error → severity: "ERROR"`; 不预留 `info/debug`（本 phase 不动 info） |
| B2 | 同 B1 + 加 `logger.info → severity: "INFO"` 预留（虽然不替换既有 console.log） |
| B3 | Cloud Logging 全 9 级 (`DEFAULT/DEBUG/INFO/NOTICE/WARNING/ERROR/CRITICAL/ALERT/EMERGENCY`) 全暴露 |

**W4 倾向：B1**。本 scope 焦点是 warn/error；info 替换是 future scope（避免 scope creep）；B3 过设计。

**请 W3 拍板**：B1 OK？

#### C. Context object shape

| 选项 | 描述 |
|---|---|
| **C1**（W4 倾向） | `logger.warn(message: string, context?: Record<string, unknown>)`；context 自动序列化 (含 Error → `{message, stack}` 转换 helper)；自动 inject `{timestamp, severity, message, module, gitSha, ...context}` 顶层 |
| C2 | `logger.warn({message, ...context})` 全对象参数 | ❌ caller signature 变（既有 console.warn 第一参是 string）|
| C3 | `logger.warn(format, ...args)` printf-style | ❌ 不结构化 |

**W4 倾向：C1**。signature 接近 `console.warn(msg, err)` 心智模型；自动 Error→object 序列化是关键便利（避免 caller 自己 `{err: e.message}` 重复）。

**请 W3 拍板**：C1 OK？Error 序列化 helper 需要含 `cause` 链吗（W1 P5.1.a-3 followup 已用 cause）？

#### R1. helper signature 不匹配既有 console.warn caller call site

- **风险**：某些 caller 用 `console.warn(msg, e, otherArg)` 多 args 模式（非 `(msg, ctx)` 模式）
- **兜底**：grep + 人审 41 lib + 30 app call sites；每个 swap commit 验该 file vitest case 全绿；overload signature: `logger.warn(msg, errorOrContext?, extraContext?)` 兼容 2-3 args

#### R2. JSON stringify 失败 (循环引用 / large object)

- **风险**：context object 含循环引用（如 Error.cause 链）→ `JSON.stringify` throw
- **兜底**：helper try/catch JSON.stringify，fallback `console.log(JSON.stringify({severity, message, error: "[serialization failed]"}))`；test case 覆盖

#### R3. P5.8.2 rebase 后 maxDuration 删 + console swap 同 file 触发不必要 merge friction

- **风险**：W2 P5.5 删 maxDuration 是简单 1 行 / route；W4 P5.8.2 swap console 是新 import + 1+ 行 swap / route；两改动行号相近可能 conflict

#### R4. structured logger 改 Cloud Logging severity 但 dev grep 难看

- **风险**：dev 跑 `npm test` / `npm run dev` 看到 raw JSON 行不如 plain text 直观
- **兜底**：E1 verdict approve 时接受 dev 体验降级；如果 W3 选 E2 (env-detect pretty-print) 接受复杂度增加
