## §1 Applicability（适用范围）

**改造目标**：把 `@vercel/blob` 7 处调用全部抽到 `lib/storage/`，对外暴露平台中性 API。
两阶段切换：

- **P5.1.a**（本 scope 主体）：起 `lib/storage/` 作为 `@vercel/blob` 的 **薄包装**
  （内部仍 import `@vercel/blob`），把 7 处 caller 全部改成 import 新 lib。
  此阶段 zero 行为变化，三门绿即可 ship。**目的**：解耦 caller 与平台，建立 seam。
- **P5.1.b**（本 scope §2.4 阶段 5）：把 `lib/storage/` 内部实现切到
  `@google-cloud/storage`。caller 零改动。**目的**：完成平台切换。

P5.1.a 解耦能落地后，P5.1.b 可以独立测试 / 灰度 / rollback 而不动 caller。

**不在本 scope**：
- P5.2 Dockerfile + Cloud Run service yaml → P5.2 scope
- P5.3 Cloud Scheduler + OIDC（替换 Vercel Cron）→ P5.3 scope
- `GCS_PRESET` 放 `lib/url-allowlist/presets.ts` → 跟 P5.2 一起（caller 路由侧改动）
- W2 phase 3.5 caller wiring（rate-limit / url-allowlist）→ 已 ship 给 W2
- Secret Manager rotate（4 secrets）→ P5.6 scope

### §2.2 GCS 调用点 → SDK API 对应表

> Vercel Blob 现有 API surface → `@google-cloud/storage` v7+ 等价调用。所有调用在
> `lib/storage/api.ts` / `lib/storage/signed-upload.ts` 内部转换，caller 看到的接口
> **完全保持** `@vercel/blob` 名字（`put`/`head`/`list`/`del`），最小化 wiring diff。

| Caller | `@vercel/blob` 调用 | `@google-cloud/storage` 等价 | 兼容性细节 |
|---|---|---|---|
| `lib/topic-cache/blob-cache.ts:36` | `head(key)` → `{url}` | `bucket.file(key).getMetadata()` + 拼 public URL | head miss → `null`（GCS 404 等价 catch） |
| `lib/topic-cache/blob-cache.ts:60` | `put(key, body, {access:"public", allowOverwrite:true, addRandomSuffix:false})` | `bucket.file(key).save(body, {contentType, public:true, resumable:false})` | `addRandomSuffix` 我们自己实现（生成随机后缀拼 key） |
| `lib/trending/snapshot-store.ts:20` | `head` | `getMetadata()` + URL | 同上 |
| `lib/trending/snapshot-store.ts:48` | `list({prefix, limit:52})` → `{blobs:[{pathname,url}]}` | `bucket.getFiles({prefix, maxResults:52, autoPaginate:false})` | 返回 `File[]`，map 成 `{pathname:f.name, url:publicUrl(f)}` |
| `lib/trending/snapshot-store.ts:88,93` | `put` retry-once | `bucket.file(key).save()` retry-once | 行为保留 |
| `lib/trending/snapshot-store.ts:109` | `list` + `del(urls[])` | `getFiles` + `Promise.all(files.map(f => f.delete()))` | `del` 接受 URLs，内部反查 key（URL 形如 `https://storage.googleapis.com/bucket/key`） |
| `lib/account-profile/cache.ts:40` | `head` | 同上 | |
| `lib/account-profile/cache.ts:55` | `put` | 同上 | |
| `app/api/compile-capcut/route.ts:156` | `put(key, Buffer, {access:"public", contentType:"application/zip", addRandomSuffix:true})` → `{downloadUrl}` | `bucket.file(key+suffix).save(Buffer, {contentType, public:true})` + 返回 signed download URL（含 `response-content-disposition=attachment`） | **downloadUrl 是 P5.1 难点**：Vercel Blob 默认带 `?download=1` 让浏览器下载；GCS 必须用 signed URL with `responseDisposition=attachment` 头，TTL 15 min |

### §2.3 决策点（5 + cross-cutting）

> 每个决策都给倾向 + reason，⏳ 等 W3 verdict。

#### A. lib API：singleton vs per-request

| 选项 | 描述 | 取舍 |
|---|---|---|
| **A1**（W1 倾向） | singleton `getStorage()` 缓存 `new Storage()` 实例（类比 `lib/technique-matching/match-engine.ts:25` 的 Anthropic SDK pattern） | ✅ Cloud Run 容器复用、ADC 一次解析；⚠️ 测试要 reset singleton |
| A2 | per-request `new Storage()` | ✅ 无全局态；❌ Cloud Run 跨请求每次重建 GCS HTTP client 浪费连接池 |

**W1 倾向：A1**。Cloud Run 容器复用 + ADC 一次解析 + 与既有 SDK pattern 一致（Anthropic / Apify
都是 lazy singleton）。test reset 用 `vi.resetModules()`，既有 match-engine test 也是这套。

#### B. 错误处理与 `@vercel/blob` API mapping

| 选项 | 描述 | 取舍 |
|---|---|---|
| **B1**（W1 倾向） | head 404 → `null`；put 失败 → throw；list 失败 → throw；del 失败 → throw | ✅ 与现有 caller 容错完全一致（caller 已 `try/catch`，head miss 走 null 路径） |
| B2 | 全部 throw 统一 `StorageError` 子类（NotFoundError / WriteError 等） | ✅ 类型更精细；❌ 7 处 caller 都要改 `catch` 分支 |

**W1 倾向：B1**。**关键约束**：caller 现状对 `head` 失败处理是 `catch → return null`
（topic-cache:41、snapshot-store:31、account-profile:45），任何把 404 改成 throw 的方案都
扩散到 caller 改造。B1 在 `lib/storage/api.ts` 内部 catch `err.code === 404 || err.message.includes('No such object')`
统一返回 `null`，保持 caller 0 改动。

#### C. signed URL TTL

| 用途 | W1 倾向 | reason |
|---|---|---|
| compile-capcut downloadUrl（zip 下载） | **15 min** | 用户点 export → 立即下载；客户端可能慢网，15 min 留足重试 |
| upload route client-direct（PUT 签名） | **60 min** | 用户上传 100-200MB 视频 / PDF，可能选完 BGM/参考素材才传，留 60 min |
| 内部 snapshot JSON / cache JSON 读取 | **N/A — 走 public URL** | bucket 公开读，无 TTL；与 Vercel Blob `access:"public"` 等价 |

**W1 倾向：以上 3 档**。⏳ W3 verdict 决定。

#### D. GCS bucket region & class

| 维度 | W1 倾向 | reason |
|---|---|---|
| region | **us-central1**（与 Cloud Run service 同区） | 同 region 出网免费（egress 0 USD），延迟 < 5ms |
| storage class | **Standard** | snapshot / cache 写完 1-7 天内频繁读；冷存（Nearline/Coldline）按 30/90 天 minimum 计费不划算 |
| Uniform bucket-level access | **Enabled** | GCS 2026 默认；object ACL 已 deprecated |
| Public read | **bucket-level + 单 bucket** | viral-reviewer-prod 一个 bucket 全公开；private 数据本来就不写 |
| Versioning | **Disabled** | snapshot/cache 是周键写覆盖，版本无意义；compile-capcut zip 已 addRandomSuffix 避撞 |

**W1 倾向：以上 5 项**。⏳ W3 verdict。

#### E. Authentication（SA key vs Workload Identity）

| 场景 | W1 倾向 | reason |
|---|---|---|
| Cloud Run runtime（service → GCS） | **Workload Identity / 默认 service account** | 不存 key，cloud-native 主流 |
| GHA push（CI → Cloud Run / GCS） | **Workload Identity Federation OIDC**（沿用 P5 verdict E1 GHA） | 不存 SA key in GitHub Secret |
| 本机 dev（开发 → GCS emulator OR sandbox bucket） | **gcloud ADC**（`gcloud auth application-default login`） | 不污染生产 key；fakeGCS emulator 不需要 |
| 测试（CI / vitest） | **GCS emulator**（fsouza/fake-gcs-server Docker） | 无网络 + 无 key + 完全离线 |

**W1 倾向：以上 4 场景**。⏳ W3 verdict。

#### Cross-cutting

| # | 决策 | W1 倾向 | reason |
|---|---|---|---|
| F | bucket CORS | **POST/PUT/GET from `https://viral-reviewer.com` + preview domain glob** | client-direct upload 必须配 CORS，否则浏览器 PUT 走不通 |
| G | bucket lifecycle GC | **暂不设（P5.1 不做）** | snapshot 已有 `pruneOldSnapshots(keepWeeks=8)`，compile-capcut zip 走 P5 verdict F1 "每周日 GC 14 天前" cron（P5.7 实施） |
| H | key naming | **完全保持现 scheme**（`topic-cache/...`, `trending/snapshot-...`, `account-profile/...`, `capcut-exports/...`） | 减小 caller / fixture 改动；本 scope 是 lib swap 不是数据 reshape |
| I | `addRandomSuffix` 实现 | **lib/storage/api.ts 内部用 `crypto.randomUUID().slice(0,8)` 拼 key** | `@vercel/blob` 默认行为是 6-8 chars base32，我们用 8 chars hex 等价 |
