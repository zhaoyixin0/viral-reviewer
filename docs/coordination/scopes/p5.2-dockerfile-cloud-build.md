### §2.3 设计决策点（A-I + J 条件性 + K 衍生）

#### A. Docker build cache 策略

| 选项 | 描述 | 取舍 |
|---|---|---|
| **A1**（W2 倾向） | multi-stage 分层 cache：`COPY package*.json` → `npm ci --omit=dev` 单 layer + `COPY . .` → `next build` 另一 layer；GHA 用 `docker/build-push-action` 内置 layer cache（push to GHA cache） | ✅ 标准模式；包 / 源码 lock 改动才重建对应 layer；GHA 免配置；✅ npm ci 是 deterministic（package-lock.json freeze） |
| A2 | BuildKit cache mount：`RUN --mount=type=cache,target=/root/.npm npm ci`；GHA 用 buildx + `cache-to type=gha,mode=max` | ✅ npm cache 跨 build 复用更高效；❌ 需 buildx 显式配置；❌ Cloud Build 不原生支持（GHA OK） |
| A3 | 不 cache（每次 fresh） | ✅ 简单；❌ 每次 npm ci 60-90s + next build 30-60s，CI 时间不可接受 |

**W2 倾向：A1**。理由：viral-reviewer 用 `package-lock.json`（已确认仓库根存在，npm 包管理），`npm ci` deterministic；`docker/build-push-action@v5` 默认 inline GHA cache export；A2 收益（npm cache reuse）相对小（npm ci 在 cache 命中 layer 时本身就 skip），增加 BuildKit 配置复杂度不划算；A3 直接放弃 CI 时间预算（每 PR build 多 1 分钟 × N PR）。

**请 W3 拍板**：A1 vs A2？是否预留未来切 A2 的退路（GHA workflow 注释 TODO）？

#### B. ffmpeg-static / ffprobe-static binaries 怎么 COPY 进 runner stage

| 选项 | 描述 | 取舍 |
|---|---|---|
| **B1**（W2 倾向） | runner stage `COPY --from=deps /app/node_modules/ffmpeg-static/ffmpeg /app/node_modules/ffmpeg-static/ffmpeg` + `COPY --from=deps /app/node_modules/ffprobe-static/bin/linux/x64/ffprobe /app/node_modules/ffprobe-static/bin/linux/x64/ffprobe`（保留 node_modules 路径） | ✅ `ffmpeg-static`/`ffprobe-static` 的 default export 仍 require-resolve 工作，**0 caller 改动**（`lib/video/ffmpeg.ts` 和 `lib/video/ffprobe-meta.ts` 现状不动）；✅ 体积最小（不带 full node_modules，只两个 binary） |
| B2 | runner stage `COPY` binaries 到 `/usr/local/bin/ffmpeg` + `/usr/local/bin/ffprobe`，并 `ENV FFMPEG_PATH=/usr/local/bin/ffmpeg` + `ENV FFPROBE_PATH=/usr/local/bin/ffprobe` | ✅ 系统标准位置，PATH 直接找；❌ 需 caller 改读 `process.env.FFMPEG_PATH` || 现有 default（小 diff 但跨 lib 影响）；❌ 失去 ffmpeg-static 包封装的 platform-detect 能力 |
| B3 | runner stage `apt-get install -y ffmpeg`（base bookworm-slim 默认 apt 可装系统 ffmpeg） | ✅ 不依赖 node_modules；✅ 装的是 distro 维护的稳定版本；❌ image size +80MB（system ffmpeg 含全 codec）；❌ 版本与 ffmpeg-static 不同步可能引入行为差异；⭐ **后备方案**：若 B1 在 bookworm-slim 上跑出 glibc 兼容问题用 B3 |

**W2 倾向：B1**。理由：当前 `lib/video/ffmpeg.ts` + `lib/video/ffprobe-meta.ts` 直接 `import ffmpegPath from "ffmpeg-static"`（runtime 解析 binary 路径），保留 node_modules 路径意味着 zero caller change；image size 增量仅 ~30MB（两个 binary）；B3 是 fallback 路径（pre-commit verify 若 fail 才切）。

**请 W3 拍板**：B1 OK？是否在 §2.6 R1 显式列 B3 fallback 触发条件（如：`docker run` smoke test `/api/analyze-video` 报 GLIBC 错误时立即切 B3）？

#### C. Cloud Run service config：declarative `service.yaml` vs imperative `gcloud run deploy` flags

| 选项 | 描述 | 取舍 |
|---|---|---|
| **C1**（W2 倾向） | `service.yaml` declarative + GHA workflow 调 `gcloud run services replace service.yaml --region=us-central1` | ✅ GitOps 友好（service config diff 进 PR review）；✅ env binding / min-instances / timeout 等参数集中可见；✅ rollback 是 `git revert` + replace；⚠️ schema 学习曲线（但 GCP docs 完备） |
| C2 | GHA workflow 直接 `gcloud run deploy SERVICE --image=... --region=... --min-instances=1 --timeout=3600 --max-instances=10 --update-secrets=... --port=8080` 等 flags | ✅ 一行命令上手快；❌ flags 跨多行 shell 字符串，可读性差；❌ 关键配置（如 secret binding）藏在 shell 命令里，code review 难 |
| C3 | Terraform module（`google_cloud_run_v2_service` resource） | ✅ 最声明式 + state diff；❌ viral-reviewer 没 Terraform 投入 + state 存储 / lock backend 设置成本；❌ 团队不熟 Terraform |

**W2 倾向：C1**。理由：`service.yaml` 是 GCP 官方推荐的 GitOps 入口（[Cloud Run 文档](https://cloud.google.com/run/docs/reference/yaml/v1)），关键配置（secret 绑定 / scaling / resource limits）一目了然；rollback 流程清晰（revert YAML + replace）；C3 对单 service 的 viral-reviewer 是 over-kill。

**请 W3 拍板**：C1 OK？还是因为初期复杂度小先 C2、后续切 C1？

### §2.6 风险面 + 兜底（含 cross-check §4 8 anti-patterns）

#### R1. ffmpeg-static binary 在 `node:24-bookworm-slim` 上 glibc 兼容性

- **风险**：W3 P5 verdict 额外 4 已 confirm bookworm-slim glibc 兼容 ffmpeg-static；但 `ffprobe-static` 包的 binary path 是 `node_modules/ffprobe-static/bin/linux/x64/ffprobe`（双重间接），COPY 路径写错可能 runtime missing
- **兜底**：短期：commit 1 pre-commit verify `curl /api/analyze-video` 实测；中期 fallback：若 B1 fail 切 B3（apt-get install ffmpeg + caller 改读 `process.env.FFMPEG_PATH=/usr/bin/ffmpeg`，但跨文件改动，需 W3 verdict 扩 ownership）；长期：监控 Cloud Logging ffmpeg 错误率

#### R2. Cloud Run cold start latency vs Vercel Fluid Compute

- **风险**：min-instances=0 时冷启动 3-5s；Vercel Fluid Compute 实例复用近零冷启动
- **兜底**：service.yaml 配 `--min-instances=1` prod 恒定 1 实例 ~$5/月（per P5 verdict 不阻塞建议 #4）；preview revisions 留 min=0；`--cpu-boost` 启动期 cpu 倍增减启动延迟

#### R3. Docker image size 失控（target < 500MB）

- **风险**：Next.js standalone + node_modules（生产 deps）+ ffmpeg/ffprobe binary 累计可能超 500MB（pdf-parse / ai sdk 等大 deps）
- **兜底**：`.dockerignore` 排除清单 §2.3 H 严格；commit 1 pre-commit verify size 实测；若 > 500MB 触发审视：换 `node:24-bookworm-slim` 而非 `node:24` full image（slim 节省 ~700MB）；ffmpeg-static 只 COPY linux/x64 binary（不带其他平台）

#### R4. WIF setup 一次性 ops 步骤错误

- **风险**：runbook §2.3 E 步骤 user 跑错（如 WIF principalSet binding `attribute.repository` 不匹配 → GHA 拿不到 SA token → deploy 全 fail）
- **兜底**：runbook 含 verify step `gcloud iam service-accounts get-iam-policy <SA>` 查 binding；GHA workflow 第一步 `google-github-actions/auth@v2` fail 时输出明确 error；初次 deploy 在 staging project 跑通后切 prod

#### R5. Cloud Run service.yaml schema 跨版本 drift

- **风险**：GCP `service.yaml` v1 schema 随 Cloud Run 更新可能弃用字段；replace 命令 fail
- **兜底**：runbook 列 `gcloud run services replace --dry-run` 在本机校验；GHA workflow 加 `gcloud version` 显示当前 SDK；订阅 GCP release notes（user 关注）

#### R6. Artifact Registry image 累积成本

- **风险**：每 commit push 一个 image，长期累积存储成本；image 各 ~250-350MB
- **兜底**：单独 GHA workflow `artifact-registry-image-gc.yml`（**本 scope 不实施，列入 P5.8 observability scope 候选**）按 keep-last-50 SHA + 14 天前 untagged 删；监控 Cloud Billing 用量

#### R7. Cloud Run startup probe 误判 ready

- **风险**：决策 J 选 J2（TCP probe only）时，容器 listen 但 Next.js 仍在初始化（如解析 vercel.ts / 加载大 deps），probe 返 ready 但实际首请求 500
- **兜底**：建议选 J1（HTTP probe `/api/health`）；若 W3 选 J2，service.yaml `startupProbe.initialDelaySeconds=10`（启动后等 10s 再 probe）+ `periodSeconds=5` + `failureThreshold=12`（总 70s 等待 Next.js ready）
